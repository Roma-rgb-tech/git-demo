# Git Merge Conflict — Конспект

## 1. Що таке merge conflict?

**Merge conflict** виникає, коли Git не може автоматично об'єднати зміни з двох гілок, бо обидві змінили **одні й ті самі рядки** одного файлу (або одна гілка видалила файл, а інша його змінила).

---

## 2. Коли виникає конфлікт?

| Ситуація | Конфлікт? |
|---|---|
| Різні файли змінені в обох гілках | ❌ Нема |
| Один файл, але різні рядки | ❌ Нема |
| Один файл, **одні й ті самі рядки** | ✅ Є конфлікт |
| Файл видалено в одній, змінено в іншій | ✅ Є конфлікт |

---

## 3. Як виникає конфлікт — покроково

```bash
git checkout main
git merge feature-branch
# → CONFLICT (content): Merge conflict in index.js
# → Automatic merge failed; fix conflicts and then commit the result.
```

---

## 4. Маркери конфлікту у файлі

Git вставляє в конфліктний файл спеціальні маркери:

```
<<<<<<< HEAD
console.log("Привіт з main");
=======
console.log("Привіт з feature");
>>>>>>> feature-branch
```

| Маркер | Значення |
|---|---|
| `<<<<<<< HEAD` | Початок змін **поточної** гілки (куди мержимо) |
| `=======` | Розділювач |
| `>>>>>>> feature-branch` | Кінець змін **гілки, що вливається** |

---

## 5. Процес вирішення конфлікту

### Крок 1 — Побачити конфліктні файли
```bash
git status
# both modified: index.js
```

### Крок 2 — Відкрити файл і вирішити конфлікт вручну

Варіанти:
- залишити зміни з `HEAD`
- залишити зміни з `feature-branch`
- написати власний варіант, що поєднує обидва

**Після вирішення** — видалити всі маркери `<<<<<<<`, `=======`, `>>>>>>>`.

### Крок 3 — Позначити конфлікт як вирішений
```bash
git add index.js
```

### Крок 4 — Завершити merge
```bash
git commit
# Git сам підставить повідомлення типу:
# "Merge branch 'feature-branch' into main"
```

---

## 6. Корисні команди під час конфлікту

```bash
git status                   # які файли в конфлікті
git diff                     # показати всі конфлікти
git merge --abort            # скасувати merge повністю
git checkout --ours index.js   # взяти версію HEAD
git checkout --theirs index.js # взяти версію гілки, що вливається
```

---

## 7. Інструменти для вирішення конфліктів

```bash
git mergetool       # відкрити налаштований merge-інструмент
```

Популярні GUI-інструменти:
- **VS Code** — вбудований merge editor (кнопки "Accept Current / Accept Incoming / Accept Both")
- **IntelliJ IDEA / WebStorm** — three-way merge viewer
- **Sourcetree, GitKraken** — візуальне вирішення конфліктів

---

## 8. Стратегії злиття (`git merge -s`)

| Стратегія | Опис |
|---|---|
| `ort` (default) | Рекурсивне злиття, найрозумніше |
| `ours` | Завжди перемагає поточна гілка (ігнорує incoming) |
| `theirs` | Завжди перемагає вхідна гілка |
| `octopus` | Для злиття 3+ гілок одночасно |

```bash
git merge -s ours feature-branch
```

---

## 9. Як уникати конфліктів

- Часто синхронізуватися з основною гілкою (`git pull --rebase`)
- Ділити роботу на маленькі PR / commits
- Уникати змін одних файлів кількома розробниками одночасно
- Обговорювати архітектуру перед великими рефакторингами

---

## 10. Rebase vs Merge (конфлікти)

| | `merge` | `rebase` |
|---|---|---|
| Конфлікти | Вирішуються **один раз** | Вирішуються **для кожного commit** |
| Історія | Зберігається + merge commit | Лінійна, чиста |
| Команда для продовження | `git commit` | `git rebase --continue` |
| Команда для скасування | `git merge --abort` | `git rebase --abort` |

---

## 11. Швидка нотатка 

```
Конфлікт при merge:
  1. git status              → знайти файли
  2. Відкрити файл           → видалити маркери, вирішити вручну
  3. git add <file>          → позначити як вирішений
  4. git commit              → завершити merge

Хочу скасувати:
  git merge --abort

Хочу взяти тільки "нашу" версію:
  git checkout --ours <file>

Хочу взяти тільки "їхню" версію:
  git checkout --theirs <file>
```
