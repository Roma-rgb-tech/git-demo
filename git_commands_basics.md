# Конспект з Git: Basics commands 

---

## 1. Основний робочий цикл (CLI Commands)

| Команда | Опис |
| :--- | :--- |
| `git clone <url>` | Клонування віддаленого репозиторію на локальну машину. |
| `git status` | Перегляд стану робочої директорії та індексу (які файли змінені/готові до коміту). |
| `git add <file>` | Додавання змін до області підготовки (Staging Area). |
| `git commit -m "msg"` | Фіксація змін у локальній історії з описовим повідомленням. |
| `git log` | Перегляд історії комітів (хеш, автор, дата, опис). |
| `git branch` | Робота з гілками (перегляд, створення, видалення). |
| `git checkout` | Перемикання між гілками або відновлення файлів (`git switch` у нових версіях). |
| `git merge <name>` | Злиття змін з вказаної гілки в поточну. |
| `git pull` | Отримання змін з віддаленого репозиторію та їх автоматичне злиття. |

## 2. Ключові концепції 

### Що таке файл .gitignore?
**Відповідь:** Це конфігураційний файл, який вказує Git, які файли або директорії слід ігнорувати. 
* **Призначення:** Забезпечення чистоти репозиторію (ігнорування `__pycache__`, `venv/`, `.DS_Store`) та безпеки (ігнорування секретів, як-от `.env` або ключі).
* **Важливо:** Сам файл `.gitignore` має бути закоммічений, щоб правила були єдиними для всієї команди.

### Гілкування та Pull Requests
**Відповідь:** * **Git Branch:** Дозволяє ізольовано працювати над фічами, не ламаючи стабільний код у `main`.
* **Pull Request (PR):** Це механізм у системах на кшталт GitHub/GitLab, який дозволяє розробнику запропонувати свої зміни для злиття в основну гілку. PR є точкою проведення Code Review та автоматичних запусків CI-тестів.

## 3. Скасування змін: Revert vs Reset

**Відповідь:**
* **Git Revert:** Створює новий коміт, що скасовує зміни попереднього. Це **безпечний** спосіб для публічних гілок, оскільки він не змінює історію.
* **Git Reset:** Переміщує вказівник гілки на певний коміт у минулому. Це **деструктивна** операція, яка переписує історію (використовується переважно локально до виконання `push`).

---

## 4. Професійні практики (Best Practices)

* **Atomic Commits:** Кожен коміт має містити одну логічно завершену зміну.
* **Merge Conflicts:** Ситуація, коли Git потребує ручного втручання через редагування одних і тих самих рядків. Вирішується розробником з наступним `git add` та `git commit`.
* **Commit Messages:** Повідомлення мають бути змістовними (наприклад, `feat: add docker-compose for dev env` замість `fix error`).
